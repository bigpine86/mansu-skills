Mansu AgenticDev Manual

Mansu AgenticDev는 AI 에이전트와 함께 소프트웨어를 계획하고, 만들고, 검증하고, 출시하기 위한 오케스트레이션 시스템입니다.

개발 프로세스

Mansu의 개발 프로세스는 AI 에이전트가 더 자율적으로 코드를 만들 수 있도록 일을 정의, 계획, 구현, 검증, 리뷰, 배포 순서로 나누는 흐름입니다. 원인 불명 문제가 생기면 mansu-debug로 잠시 멈춰 원인을 먼저 찾습니다.

시작점 
mansu-help
단계 의미 설명
Define 정의 무엇을 만들지 정하는 단계입니다. 해결하려는 문제, 필요한 기능, 만들 범위, 완료 기준을 명확히 합니다.
Plan 계획 정의한 내용을 실제 작업으로 나누는 단계입니다. 어떤 기능부터 만들지, 어떤 방식으로 구현할지, 필요한 작업 순서를 정합니다.
Build 구현 계획한 내용을 실제로 만드는 단계입니다. 코드를 작성하고, 화면을 만들고, 데이터 저장이나 기능 동작을 구현합니다.
Verify 검증 만든 기능이 의도한 대로 동작하는지 확인하는 단계입니다. 버튼, 입력값, 저장, 조회, 오류 상황 등을 직접 테스트합니다. 통과하면 기본적으로 Review로 이어집니다.
Review 리뷰 결과물을 다시 점검하는 단계입니다. 코드가 너무 복잡하지 않은지, 빠진 요구사항은 없는지, 사용자가 불편할 부분은 없는지 확인합니다.
Ship 배포 검증과 리뷰가 끝난 결과물을 실제 사용자가 쓸 수 있게 내보내는 단계입니다. 운영 환경에 반영하거나 릴리즈하고, 필요하면 사용 안내를 함께 제공합니다.

큰 작업을 시작하기 전에 목적, 사용자, 문제, 성공 기준, source of truth를 잡습니다. 정의가 끝나면 다음 단계로 mansu-2plan을 안내합니다.

조합하는 원천

  • Ouroboros interview/pm/seed/brownfield초기 정의 phase 안에서 목적, 제품 정의, 장기 기억, 기존 코드베이스 파악을 깊게 만듭니다.
  • gstack-office-hours사용자, 문제, wedge, MVP가 진짜인지 창업자 관점으로 압축합니다.
  • Oh My research/deepsearch해당 phase 안에서 Threads, Reddit, GitHub, 실제 레포를 뒤져 참고 프로젝트와 인사이트를 찾습니다.
  • 사용자의 아이디어와 실제 문제를 분리합니다.
  • 목표 사용자, 성공 기준, 제약, source of truth를 정리합니다.
  • 기존 코드가 있으면 brownfield로 현재 상태를 읽습니다.
  • 구현하지 않고 mansu-2plan으로 넘깁니다.

$mansu-2plan

정의된 프로젝트를 요구사항, 아키텍처, 디자인 방향, 구현 로드맵, 현재 phase 실행 계획으로 바꿉니다. 계획이 끝나면 다음 단계로 mansu-3build를 안내합니다.

조합하는 원천
  • addyosmani spec-driven-development / planning-and-task-breakdown요구사항과 실행 단위를 구현 가능한 계획으로 바꿉니다.
  • gstack-autoplan / gstack-plan-*제품, 디자인, 엔지니어링, DX 관점으로 계획을 비평합니다.
  • Open Design / design-md referencesUI가 있으면 디자인 방향과 artifact 기준을 잡습니다.
  • DOCUMENT_CREATION_ORDER.mdspec, TDR, DESIGN, active plan을 어떤 순서로 만들지 정합니다.
  • Requirements/Spec → Architecture → Design Direction → Implementation Roadmap 순서로 잡습니다.
  • 큰 phase map과 정확히 하나의 현재/다음 phase를 고릅니다.
  • 활성 계획은 PLAN.md뿐 아니라 .omo/plans/*.md 같은 경로도 인정합니다.
  • slice boundaries, validation gates, risks를 명확히 한 뒤 mansu-3build로 넘깁니다.

$mansu-3build

활성 phase 실행 계획을 기준으로 실제 구현에 들어갑니다. 내부 구현 엔진은 mansu-tdd-total입니다. 구현이 끝나면 다음 단계로 mansu-4verify를 안내합니다.

조합하는 원천
  • mansu-tdd-total활성 계획을 검토하고 순차 vertical slice 실행으로 연결합니다.
  • mansu-tdd-lite / mansu-tdd-strict각 slice를 가벼운 검증 중심으로 갈지, RED 테스트부터 잠글지 실행 방식을 맡습니다.
  • addyosmani planning-and-task-breakdown큰 작업을 실행 가능한 작은 작업 단위로 쪼개는 관점을 보강합니다.
  • CODE_CONSTRUCTION_ORDER.md현재 slice의 구현 단계와 참고할 원천 스킬을 고릅니다.
  • gstack-guard / gstack-careful / checkpoint routes구현 중 안전선과 중간 증거를 남깁니다.
  • PLAN.md, Plan.md, .codex/plans/*.md, .omo/plans/*.md, 사용자가 가리킨 계획 파일을 먼저 찾습니다.
  • 파일명이 다르다는 이유만으로 계획이 없다고 말하지 않습니다.
  • 작업을 순차 vertical slice로 나눕니다.
  • 각 slice의 완료 조건, 영향 파일, 테스트 기준, 위험 요소를 정합니다.
  • 필요하면 내부적으로 strict/lite 방식을 선택합니다.

$mansu-debug

증상만 덮지 않고, 다시 깨지지 않게 원인을 잠급니다.

조합하는 원천
  • Oh My debug버그를 증상, 재현, 원인, 수정, 검증으로 밀고 가는 기본 디버그 루프입니다.
  • Oh My doctor / build-fix / git-master환경 문제, 빌드 실패, 회귀 시작점 추적처럼 특수한 문제를 맡습니다.
  • gstack-investigate근본 원인, scope lock, 패턴 분석, 회귀 증거를 강하게 요구하는 조사 게이트입니다.
  • gstack-browse / gstack-qa-only브라우저에서 재현 증거, 스크린샷, 콘솔, 사용자 플로우를 수집합니다.
  • gstack-benchmark느림, 성능 회귀, 리소스 증가가 의심될 때 기준선을 잡습니다.
  • 재현 경로와 증거를 먼저 확보합니다.
  • 원인을 격리하고 가설을 세웁니다.
  • 최소 수정으로 고칩니다.
  • 회귀 테스트 또는 수동 검증 증거를 남깁니다.
산출물: 근본 원인산출물: 수정 + 회귀 증거

$mansu-4verify

구현이 실제로 맞는지 테스트, 런타임, 브라우저, QA evidence로 증명합니다. 통과하면 기본적으로 mansu-5review를 이어 실행합니다.

조합하는 원천
  • mansu-web-verify웹앱 검증이 필요할 때 쓰는 compatibility route입니다.
  • gstack-browse실제 브라우저를 열어 버튼, 폼, 화면 상태, 콘솔을 확인합니다.
  • gstack-qa-only수정 없이 QA 리포트, 재현 단계, 위험도를 남깁니다.
  • gstack-benchmark로드 속도, 체감 지연, 리소스 크기를 성능 관점으로 봅니다.
  • gstack-cso입력 검증, 권한, 비밀정보, 공급망 같은 보안 리스크를 봅니다.
  • Oh My playwright / ultraqa / security-review검증을 반복 가능한 E2E, 넓은 QA, 간단 보안 체크로 확장할 때 씁니다.
  • 테스트, 실행 결과, 브라우저 확인을 증거로 남깁니다.
  • 수정 없이 현재 동작을 먼저 증명합니다.
  • 검증이 통과하면 mansu-5review로 이어가고, 실패하거나 증거가 부족하면 멈춥니다.
  • 문제를 재현 가능한 형태로 정리하고 필요하면 mansu-debug로 넘깁니다.
산출물: QA evidence산출물: 재현 단계와 위험도산출물: review handoff

$mansu-5review

검증된 변경을 품질, 구조, 보안, 디자인, 유지보수, 의사결정 리스크 관점으로 판단합니다. Review는 QA를 대체하지 않습니다. 리뷰가 통과하면 다음 단계로 mansu-6ship을 안내합니다.

조합하는 원천
  • gstack-reviewdiff 관점에서 회귀, 누락 테스트, 위험한 side effect를 찾습니다.
  • gstack-design-reviewUI 변경의 시각 품질과 상호작용 완성도를 봅니다.
  • addyosmani code-review-and-quality리뷰 품질과 유지보수 기준을 보강합니다.
  • 검증 evidence가 있는 변경만 review합니다.
  • severity별 finding과 required fixes를 남깁니다.
  • 원인이 불명확한 문제는 mansu-debug로 넘깁니다.
산출물: review findings산출물: required fixes산출물: reverify route

$mansu-6ship

검증과 리뷰 뒤 실제로 내보낼 수 있는지 ship/hold를 판단합니다. release 실행은 필요할 때 mansu-ship-release로 연결합니다.

조합하는 원천
  • gstack-health출하 전 테스트, 타입, 린트, 품질 신호를 종합 점검합니다.
  • gstack-reviewdiff 관점에서 회귀, 누락 테스트, 위험한 side effect를 찾습니다.
  • mansu-ship-release커밋, 푸시, PR, changelog 같은 release readiness compatibility route입니다.
  • gstack-ship실제 PR/출하 루틴을 맡습니다.
  • gstack-document-releaseREADME, CHANGELOG, 아키텍처 문서가 실제 변경과 맞는지 동기화합니다.
  • gstack-land-and-deploy / gstack-canary머지, 배포, 배포 직후 감시를 담당합니다.
  • gstack-retro / gstack-learn이번 작업의 교훈과 재사용할 패턴을 남깁니다.
  • verify evidence와 review/risk notes가 있는지 확인합니다.
  • health, QA, docs 상태를 확인합니다.
  • 보안/성능/배포 리스크가 있으면 추가 게이트를 겁니다.
  • 출하/보류 판단과 남은 리스크를 정리합니다.

  • 재현과 증거 수집증상, 재현 경로, 브라우저/로그 증거를 먼저 확보합니다.
  • 격리와 가설scope lock, pattern analysis, root cause hypothesis를 세웁니다.
  • 최소 수정과 회귀 증명필요 최소한만 고치고 같은 문제가 재발하지 않는 증거를 남깁니다.
  • 목적: 원인 없는 임시 봉합을 막고, 다시 깨질 확률을 낮춥니다.
  • 언제: 버그, 회귀, stack trace, 이상 동작, 환경/빌드 문제를 다룰 때 씁니다.
  • 주요 결과: root cause와 수정 근거, 회귀 증거가 남습니다.
산출물: root cause산출물: minimal fix산출물: regression proof

$mansu-web-verify

내부에서 하는 일
  • 플로우 탐색페이지, 버튼, 폼, 반응형, 콘솔 상태를 실제 브라우저에서 확인합니다.

  • 목적: 구현이 끝난 뒤 사용자 관점에서 어떤 문제가 남았는지 드러냅니다.

  • 주요 결과: 재현 가능한 QA 보고서와 위험 요약이 생깁니다.

에이전트를 위한 Mansu 사용법

Mansu는 모든 걸 직접 구현하는 스킬이 아니라, 현재 작업에 맞는 원천 스킬을 고르고 조합하는 오케스트레이터입니다.

0. 매뉴얼은 실행 지침이 아니라 루트 지도다

특정 Mansu 스킬이 이미 선택됐으면 그 스킬의 SKILL.md가 실행 기준입니다. 이 탭은 모든 스킬이 매번 읽는 필수 문서가 아니라, 어떤 Mansu 루트를 골라야 할지 애매할 때 보는 지도입니다.

주로 mansu-0help, mansu-9setup, mansu-0manual, mansu-source-curator 같은 입구/세팅/관리 흐름에서 참고합니다.

1. 먼저 상황을 분류한다

사용자가 막연하면 바로 구현하지 말고 mansu-0help 관점으로 현재 상황을 분류합니다.

  • 설치/업데이트: mansu-9setup
  • 새 아이디어/새 앱/새 프로젝트/큰 기능군: mansu-1define, 그다음 mansu-2plan
  • 구현: 현재 phase 실행 계획이 있으면 mansu-3build, 없으면 먼저 mansu-1define/mansu-2plan
  • 동작 증명/QA: mansu-4verify
  • 품질/구조/보안/디자인 리뷰: mansu-5review
  • 버그: mansu-debug
  • 출하 판단: mansu-6ship

2. 직접 흉내 내지 말고 연결한다

Mansu의 핵심은 좋은 원천 스킬을 재구현하지 않는 것입니다. 강한 원천 루트가 있으면 그 루트를 읽거나 호출하고, Mansu는 판단과 연결을 맡습니다.

  • 전체 phase spine: addyosmani/agent-skills
  • 초기 정의/기억/seed: Ouroboros 계열
  • phase 안 실행 모드: Oh My 계열
  • 전문가 비평/QA/출하 게이트: gstack 계열

3. 설치는 실행 환경에 맞춘다

mansu-9setup은 런타임, 에이전트 호스트, OS를 먼저 확인합니다. Codex에서는 검증된 LazyCodex / OMO를 확인하고 legacy omx 호환성도 별도로 봅니다. OpenCode/Hermes는 omo, Claude Code는 omc를 맞는 adapter로 봅니다.

확신이 낮으면 추측 설치하지 말고 check/report 또는 사용자 확인으로 멈춥니다.

4. 프로젝트 시작은 큰 설계부터

mansu-1define은 목적, 사용자, 요구사항, 성공 기준, source of truth를 잡고, mansu-2plan은 spec/TDR, UI 방향, project roadmap/phase order와 현재 phase를 정합니다. mansu-project-start는 이 흐름의 compatibility alias입니다.

구현 handoff 전에는 목적, 사용자/문제, 근거, 가정, 방향, 순서, 실행 연결이라는 Zero-to-PLAN minimum gate를 통과해야 합니다. 위험이 작으면 짧게 기록하고, 위험이 크면 리서치/spec/TDR/DESIGN/ADR로 확장합니다.

초기 정의는 Ouroboros interview/pm/seed로 깊게 만들고, phase 안 리서치는 Oh My research, 비평과 검증은 gstack office-hours/autoplan/plan-*로 조합합니다.

전체 기능 순서는 spec/TDR/design layer에 남기고, PLAN.md는 현재 phase의 실행 문서로 둡니다. 그 phase의 세부 slice, 구현, review, QA, checkpoint는 mansu-3build가 맡고 내부 구현 엔진으로 mansu-tdd-total을 사용합니다.

5. 구현은 build에서 시작한다

사용자가 구현을 원하고 현재 phase 실행 계획이 있으면 기본은 mansu-3build입니다. PLAN.md만 보지 말고 Plan.md, .codex/plans/*.md, .omo/plans/*.md, 사용자가 가리킨 계획 파일까지 확인합니다.

mansu-3build가 내부 엔진으로 mansu-tdd-total을 사용해 계획을 비평하고 순차 vertical slice를 만든 뒤 각 slice를 lite/strict로 연결합니다.

  • 다음 slice는 이전 slice가 닫힌 뒤 시작합니다.
  • 각 slice는 완료 조건, 영향 파일, 테스트 기준, 위험 요소를 가져야 합니다.
  • 작업 기록은 PLAN.md와 worklog에 남깁니다.

6. 버그는 원인 전 수정 금지

버그/회귀/stack trace는 mansu-debug-rootcause로 다룹니다. 재현, 격리, 패턴 분석, 가설, 최소 수정, 회귀 증거가 기본 순서입니다.

UI 증거는 gstack browse/qa-only를 쓰고, 수정 포함 QA는 근본 원인 확인 뒤 또는 사용자가 명시적으로 원할 때만 씁니다.

7. 마지막 보고는 사용자 언어로

최종 보고는 사용자의 언어 또는 프로젝트의 기본 언어를 따릅니다. 무엇을 바꿨는지, 어떤 검증을 했는지, 남은 리스크나 추가 확인이 필요한지를 짧게 남깁니다.

구현 작업은 slice가 닫힐 때마다 checkpoint와 커밋을 남기는 것이 기본입니다. 단, 도움말/점검/매뉴얼 표시처럼 변경 없는 작업은 커밋하지 않고, push/PR/deploy는 사용자가 요청했거나 출하 흐름이 명확할 때만 진행합니다.

8. 에이전트/관리자용: 매뉴얼도 산출물이다

이 규칙은 Mansu를 유지보수하는 에이전트와 관리자에게 적용됩니다. Mansu 스킬, 원천 스킬, 버전, 날짜, 루트 의미가 바뀌면 이 HTML 매뉴얼과 validator도 같이 갱신합니다.

mansu-9setup source-check는 개발 프로세스 단계가 아니라 관리자용 source freshness 점검입니다. drift가 확인되면 승인 후 mansu-source-curator와 매뉴얼/validator 갱신으로 이어갑니다.

매뉴얼은 사람용 UI이면서 입구/관리 흐름에서 참고하는 에이전트용 루트 맵입니다.
